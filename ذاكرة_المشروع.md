# ذاكرة مشروع التفسير المحرر (Project Memory)

يوثق هذا المستند القرارات المعمارية والبرمجية، والتطور الهندسي لمشروع استخلاص وتوليد مجموعة بيانات (Dataset) موسوعة التفسير المحرر من موقع الدرر السنية.

---

## 1. الفلسفة المعمارية ومبادئ التصميم

* **تجنب التعقيد السحابي (Local-First):** الاعتماد بالكامل على معالجة البيانات محلياً (HTML إلى JSON) لضمان السيادة والاستقلالية والسرعة الفائقة.
* **الكود النظيف وقواعد التطوير:** التزام التفكيك الوظيفي الصارم للمسؤوليات (Single Responsibility Principle) والاختصار والتبسيط لضمان سهولة الصيانة والمراجعة البشرية.
* **الأمانة العلمية والدقة التقنية:** مطابقة نصوص التفسير والأحكام مع الأصول المحلية عبر بصمات SHA-256 للمتون وعزل الحواشي بربطها بمتونها برمز الإحالة دون تشويه النص.
* **إخلاء المسؤولية العلمية والتقنية:** توثيق أن المخرجات البرمجية لم تخضع لمراجعة بشرية تامة، لضمان الفهم البحثي السليم.

---

## 2. الهيكل التنظيمي للمشروع

تتوزع ملفات المشروع وسوره وفق البنية الشجرية التالية:

```text
التفسير المحرر/
│
├── tafseer html/                     # [مجلد محلي متجاهل] مجلد مستقل لكل سورة محلياً
│   ├── المقدمة.html                  # مقدمة السورة الخام
│   └── [بداية الآية]-[نهاية الآية].html   # صفحة تفسير فرعية خام (مثال: 1-5.html)
│
├── json/                             # ملفات JSON مستقلة لكل سورة
│   └── [رقم السورة].json             # ملف JSON منظّم للسورة الواحدة
│
├── scripts/                          # السكربتات والأدوات البرمجية المطورة
│   ├── downloader.py                 # سكربت التحميل التلقائي وتتبع التقدم
│   ├── helpers.py                    # دوال تصفية وتحقق ومعالجة نصوص مشتركة
│   ├── json_generator.py             # سكربت توليد JSON من الأصول المحلية
│   ├── validator.py                  # سكربت التحقق من JSON ومطابقة البصمات
│   ├── get_tafsir.py                 # أداة CLI محلية لعرض وتصفية نصوص السور والآيات
│   └── run_tests.py                  # سكربت الاختبارات المؤتمت الشامل للأدوات
│
├── progress.json                     # [ملف محلي] تتبع تقدم التحميل لاستئناف العمل
├── هيكل_الصفحات.md                    # توثيق الكلاسات والمعرفات الخاصة بالصفحات
├── tafseer_all.json                  # ملف JSON موحد يضم السور المكتملة
└── ذاكرة_المشروع.md                   # ذاكرة المشروع والقرارات الفنية والتصميم ومخطط JSON (هذا الملف)
```

## 3. السكربتات البرمجية ووظائفها

### أ. أداة جلب وعرض التفسير (`scripts/get_tafsir.py`)
* **الوظيفة:** أداة واجهة سطر أوامر تفاعلية لعرض التفسير وغريب الكلمات والإعراب والفوائد لسورة معينة أو نطاق محدد من الآيات مع تصفية الحواشي أو إثباتها.
* **الخصائص الفنية:**
  - تدعم التصفية التفصيلية عبر المعاملات (`--verses`, `--vocab`, `--meaning`, `--detail`, `--grammar`, `--balagha`, `--edu`, `--sci`).
  - تدعم حجب المراجع بالكامل عبر المعامل (`--no-ref`).
  - مقسمة إلى دوال فرعية مصغرة متسقة مع قواعد التطوير النظيف لسهولة القراءة والتحسين.

### ب. سكربت التحميل التلقائي (`scripts/downloader.py`)
* **الوظيفة:** يتنقل بين صفحات التفسير برمجياً بالاعتماد على روابط "التالي" لاستكشاف السور وجلبها من موقع الدرر السنية وحفظها محلياً.
* **الخصائص الفنية:**
  - يعتمد فاصل زمني لتجنب الحظر، ويحفظ الصفحات داخل مجلد `tafseer html/`.
  - يستخدم مكتبة `progress` لعرض شريط تقدم تفاعلي بالطرفية.
  - يدعم حفظ واستئناف العمل من خلال تصفح ملف `progress.json`.

### ج. مكتبة الدوال المشتركة (`scripts/helpers.py`)
* **الوظيفة:** تجمع دوال معالجة النصوص العربية وتنقية HTML وتصنيف الأقسام واستخراج الحواشي والبصمات النصية المشتركة بين أدوات التوليد والتحقق والتشغيل.
* **الخصائص الفنية:**
  - تمنع تكرار منطق التصفية والتصنيف وتضمن تطابق مقارنة البصمات واستخراج المراجع.
  - تدعم الحواشي النصية العارية التي تبدأ برقم مثل `[404] رواه...` ولو لم تكن داخل وسوم HTML الخاصة بالدرر.

### د. سكربت توليد ملفات JSON (`scripts/json_generator.py`)
* **الوظيفة:** يستخلص بيانات التفسير المحرر من ملفات HTML الخام ويولد ملفات JSON المستقلة والملف الموحد.
* **الخصائص الفنية:**
  - يفرز الحقول بنيوياً كنص ظاهر نظيف مستخرج من HTML ويحفظها بمفاتيح ثابتة بالإنجليزية.
  - يعزل مقدمات السور في حقل وصفي مخصص (`introduction`).
  - يحسب البصمات النصية `section_sha256` و`introduction_sha256` للتأكد من عدم تغيّر النص عن المصدر المحلي.
  - يفصل الفوائد إلى حقلين مستقلين: `educational_benefits` للفوائد التربوية، و`scientific_benefits` للفوائد العلمية واللطائف.

### هـ. سكربت التحقق والمطابقة (`scripts/validator.py`)
* **الوظيفة:** أداة تدقيق بنيوية ومطابقة رقمية تقوم بمقارنة بصمات SHA-256 للمتون في JSON مع أصول HTML، وفحص مطابقة الحواشي والمراجع وتوزيعها، وضمان سلامة المخطط.
* **الخصائص الفنية:**
  - يتحقق من وجود الحقول المطلوبة وخلو البيانات تماماً من وسوم HTML الخام.
  - يطابق مراجع الصفحة مع المتن ويصدر تقريراً شاملاً بالتحذيرات والأخطاء.

### و. سكربت الاختبارات المؤتمتة (`scripts/run_tests.py`)
* **الوظيفة:** يختبر سيناريوهات تشغيل مختلفة لأداة الجلب `get_tafsir.py` للتحقق من سلامة الأوامر ومعالجة الأخطاء وأكواد الخروج.
* **الخصائص الفنية:**
  - ينفذ اختبارات تكاملية سريعة للتحقق من عدم حدوث تراجعات (Regressions) بعد تعديل الكود المشترك.

## 4. المبادئ الفنية والقواعد الهندسية المعتمدة

### أولاً: دقة استخراج نطاق الآيات (Verse Range Extraction)
لمنع الفشل في استخراج أسماء الملفات، تم تدوين تعبير نمطي (Regex) يدعم كافة الصياغات العربية الممكنة لتحديد نطاق الآيات في كروت الصفحات:
`r'(?:الآيات|الآية|الآيتان|الآيتين)\s*[:]?\s*[\(]?\s*(\d+)\s*(?:[-–—]\s*(\d+))?\s*[\)]?'`

### ثانياً: قواعد التصفية والتحويل للـ Markdown
1. **معالجة خطوط الآيات العثمانية:** تستخدم حاوية `div#qpage` رموز خطوط عثمانية خاصة بموقع الدرر تظهر كرموز مبهمة محلياً. يتم برمجياً البحث عن النص العربي السليم للمتن (المشكول) من داخل أول وسم `span.aaya` في جزء التفسير، ويستبدل بمحتوى `qpage` بالكامل بصيغة اقتباس منسقة (`blockquote`).
2. **إزالة التنسيق والبرمجة:** حذف وسوم `<style>` و `<script>` بالكامل عبر تفكيك شجرة الـ HTML (`.decompose()`).
3. **تطهير التبويبات الداخلية:** إزالة القوائم التبويبية والروابط الجانبية للفقرات (المحددة بكلاسات مثل `nav`, `navbar`, `collapse`, `menu`, `tabs`).
4. **حذف أزرار التصفح:** تفكيك روابط "السابق" و"التالي" التي تقع داخل كرت المحتوى.
5. **تجاهل التعليقات:** تصفية كافة كتل التعليقات البرمجية (`Comment` objects) لمنع تسربها لملفات المتن.
6. **تنظيف السطور الفارغة:** معالجة المسافات البيضاء والأسطر الفارغة الزائدة برمجياً لتجنب التكرار.

## 5. مواصفات وتصميم مخرجات JSON (JSON Specifications)

تخضع البيانات لقواعد صارمة تمنع تسرب وسوم HTML وتفصل المتون عن الحواشي:

### أولاً: قواعد النص
* **تجريد النصوص**: تُحفظ الحقول كنصوص ظاهرة خالية من وسوم التنسيق البرمجية.
* **إحالات الحواشي**: يُبقى في المتن رقم الحاشية المختصر كرمز ربط (مثال: `[15]`).
* **عزل المراجع**: تنقل النصوص الكاملة للحواشي إلى قائمة `references` الموزعة حسب الأقسام (التفسير، البلاغة، الإعراب، غريب الكلمات... إلخ).

### ثانياً: بنية ملف السورة
يُنتج التوليد نوعين من الملفات:
1. **ملفات السور الفردية**: داخل مجلد `json/` باسم `[رقم السورة].json`.
2. **الملف الموحد**: `tafseer_all.json` الذي يضم كافة السور المكتملة.

نموذج لبنية السورة الواحدة:
```json
{
  "surah_num": 112,
  "surah_name": "سورة الإخلاص",
  "introduction": "نص المقدمة مع أرقام الحواشي فقط مثل [1]...",
  "introduction_references": [
    "[1] نص الحاشية الكامل..."
  ],
  "introduction_sha256": "hash64",
  "tafseer_pages": [
    {
      "verses": "« قُلْ هُوَ اللَّهُ أَحَدٌ * اللَّهُ الصَّمَدُ »",
      "vocabulary": "نص غريب الكلمات مع أرقام الحواشي فقط أو null",
      "general_meaning": "نص المعنى الإجمالي",
      "tafseer": "نص التفسير مع أرقام الحواشي فقط",
      "grammar": null,
      "balagha": "نص البلاغة أو null",
      "educational_benefits": null,
      "scientific_benefits": "نص الفوائد العلمية واللطائف أو null",
      "references": {
        "vocabulary": [],
        "general_meaning": [],
        "tafseer": [],
        "grammar": [],
        "balagha": [],
        "educational_benefits": [],
        "scientific_benefits": []
      },
      "section_sha256": {
        "tafseer": "hash64"
      },
      "verse_range": "1-4"
    }
  ]
}
```

### ثالثاً: تفصيل حقول كائن السورة وصفحة التفسير
* `introduction_sha256` / `section_sha256`: بصمات SHA-256 لنصوص المتون بعد التنظيف لمطابقتها مع الأصول والتحقق من سلامتها.
* `vocabulary`: غريب الكلمات، أو `null` عند الغياب.
* `general_meaning`: المعنى الإجمالي، أو `null` إذا لم يوجد في HTML المصدر.
* `tafseer`: التفسير التفصيلي، أو `null` إذا لم يوجد في HTML المصدر.
* `grammar`: الإعراب، أو `null` عند الغياب.
* `balagha`: البلاغة، أو `null` عند الغياب.
* `educational_benefits`: الفوائد التربوية، أو `null` عند الغياب.
* `scientific_benefits`: الفوائد العلمية واللطائف، أو `null` عند الغياب.

### رابعاً: تصميم حواشي المراجع (References)
تكون مراجع الصفحة كائناً ثابت المفاتيح مقسماً حسب الأبواب لتسهيل العرض في طبقات مستقلة:
```json
{
  "vocabulary": [],
  "general_meaning": [],
  "tafseer": ["[15] نص الحاشية الكامل..."],
  "grammar": [],
  "balagha": [],
  "educational_benefits": [],
  "scientific_benefits": []
}
```
يرتبط الرقم المختصر داخل المتن (مثل `[15]`) بالحاشية التي تبدأ بالرقم نفسه داخل قائمة القسم المناسب في كائن `references`.

## 6. دليل تشغيل أدوات المشروع

### قاعدة عامة قبل التشغيل

شغّل الأوامر من جذر المشروع:

```powershell
Set-Location -LiteralPath "E:\GitHub\my workflows\التفسير المحرر"
```

وعند تشغيل أي سكربت يطبع العربية في الطرفية، اضبط الترميز أولاً في نفس الجلسة:

```powershell
$env:PYTHONIOENCODING="utf-8"
```

### أ. تنزيل ملفات HTML الخام

يستخدم عند الحاجة إلى تحميل الصفحات أو استكمال الناقص من المصدر:

```powershell
python "scripts/downloader.py"
```

الناتج الأساسي: مجلدات السور وملفات HTML الخام، مع تحديث `progress.json`.

### ب. توليد JSON لسورة واحدة

الوضع الآمن الافتراضي في `json_generator.py` هو سورة الإخلاص `112` إذا لم يُحدّد رقم سورة:

```powershell
python "scripts/json_generator.py" --no-progress
```

لتوليد سورة محددة:

```powershell
python "scripts/json_generator.py" --surah 84 --no-progress
```

الناتج:
* `json/84.json`
* `tafseer_all.json` ويحتوي السورة أو السور التي شملها آخر تشغيل.
* `json_generation_report.txt`

### ج. توليد JSON لكل السور المتاحة

لا يُستخدم التوليد الشامل إلا بأمر صريح:

```powershell
python "scripts/json_generator.py" --all
```

يمكن إضافة `--no-progress` عند الحاجة إلى إخراج طرفية أبسط:

```powershell
python "scripts/json_generator.py" --all --no-progress
```

### د. التحقق من سورة مستقلة

بعد توليد سورة معينة، افحص ملفها المستقل:

```powershell
python "scripts/validator.py" --surah 84
```

الناتج: `json_validation_report.txt`، وكود خروج غير صفري عند وجود أخطاء.

### هـ. التحقق من الملف الموحد

لفحص `tafseer_all.json`:

```powershell
python "scripts/validator.py" --unified
```

وإذا شُغّل `validator.py` بلا خيارات، فهو يفحص `tafseer_all.json` افتراضياً:

```powershell
python "scripts/validator.py"
```

تنبيه: لا يوجد خيار `--all` في `validator.py`؛ التحقق الشامل يكون بفحص `tafseer_all.json` بعد توليده بـ `json_generator.py --all`.

### و. تسلسل اختبار موصى به بعد تعديل الكود

```powershell
$env:PYTHONIOENCODING="utf-8"
python -m py_compile "scripts/helpers.py" "scripts/json_generator.py" "scripts/validator.py"
python "scripts/json_generator.py" --surah 84 --no-progress
python "scripts/validator.py" --surah 84
python "scripts/json_generator.py" --surah 90 --no-progress
python "scripts/validator.py" --surah 90
python "scripts/validator.py" --unified
```

للاختبار الرجعي الأوسع استعمل السور التي كشفت حالات بنيوية مختلفة:

```powershell
python "scripts/json_generator.py" --surah 80 --no-progress
python "scripts/validator.py" --surah 80
python "scripts/json_generator.py" --surah 112 --no-progress
python "scripts/validator.py" --surah 112
```

### ز. معنى التقارير

* `json_generation_report.txt`: أخطاء التوليد والقراءة والكتابة.
* `json_validation_report.txt`: أخطاء المخطط، البصمات، المراجع، ونطاقات الصفحات.
* الأخطاء (`Errors`) تمنع اعتماد المخرجات.
* التحذيرات (`Warnings`) تحتاج مراجعة، لكنها قد تكون طبيعية عند تجميع المصدر أكثر من آية في وسم واحد أو تقسيم الآية بين وسوم.
